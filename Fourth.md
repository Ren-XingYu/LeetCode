# 测试设计

```
常见的测试设计方法
	识别测试因子
		IBO模型
		思维导图
	确定因子取值
		等价类划分
		边界值分析
	确定因子组合
		All Combinations：所有取值进行全组合，组合个数=每个因子取值个数的乘积
		Basic Choices：选一个基本用例，每次只改变一个因子，组合个数=所有因子取值个数-因子个数+1
		Each Choice：每次因子至少出现一次，组合个数=最多因子的取值
		2-Wise、N-Wise：每两个因子的取值至少覆盖一次，组合个数=最多因此取值个数*次多因子取值个数
	确定预期结果
	
代码覆盖
	语句覆盖
	判断覆盖
	条件覆盖：条件覆盖不能保证判定覆盖
	判定条件覆盖：同时满足条件覆盖和判定覆盖
	组合条件覆盖：每个判定的组合执行执行一次，可能有些路径覆盖不了
	路径覆盖：不一定满足组合条件覆盖
```

# ASTRIDE LowLevel威胁建模流程

```
绘制数据流图 -> 威胁分析 -> 风险评估 -> 指定消减措施 -> 产品响应

数据流图：外部交互方、处理过程、数据存储、数据流、信任边界

Spoofing（仿冒）						认证（密码、SSL/TLS、IPSec、SSH）
Tampering（篡改）						完整性（Hash、MAC、数字签名、ACL等）
Repudiation（抵赖）						防抵赖（认证、审计日志）
Information Disclosure（信息泄露）		机密性（加密、ACL等）
Denial of Service（拒绝服务）				可用性（负载均衡、过滤、缓存等）
Elevation of Privilege（权限提升）		授权（权限最小化、沙箱等）
Privacy（隐私）							合法（匿名化、用户可知可控、数据最小化）
```

# 安全架构

安全三要素：可用性、机密性、完整性

经典设计原则：

```
开放设计
失败-默认安全
职责分离
最小权限
最小公共化（共享、连接越少越安全）
经济适用：保持系统的设计和代码尽可能简单、紧凑（简单、精巧、组件化）
完全仲裁：对每个对象的每次访问都必须经过安全检查审核
心理可承受：安全机制可能为用户增加额外的负担、这种负担必须是最小的
```

# 设计原则与设计模式

```
正交四原则
	最小化重复
	分离变化
	缩小依赖范围
	向稳定方向依赖
	
SOLID原则

简单设计的四个基本原则
	通过所有测试
	尽可能消除重复
	尽可能清晰表达
	更少的代码元素
```
